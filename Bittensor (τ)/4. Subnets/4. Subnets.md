



# Bittensor — Subnets

---

## Overview

Subnets are the **operational layer** of Bittensor. If the global network is the economy, subnets are the individual industries within it. Each subnet is a self-contained competition focused on one specific intelligence task — with its own miners, validators, rules, and emission pool.

> A subnet is a dedicated market for one type of intelligence.

Everything in the Mechanics and Incentives folders plays out _inside_ each subnet. Yuma Consensus runs per-subnet. Emissions are per-subnet. Miners and validators register per-subnet.

---

## 1. What a Subnet Is

A subnet consists of:

|Component|Description|
|---|---|
|**UID**|Unique subnet identifier on-chain (e.g. Subnet 1, Subnet 18)|
|**Subnet Owner**|The entity that created and maintains the subnet|
|**Incentive mechanism**|Custom scoring logic defining what "good output" is|
|**Miner pool**|All registered miners competing in this subnet|
|**Validator pool**|All registered validators judging in this subnet|
|**Emission allocation**|The subnet's share of global TAO emissions|
|**Hyperparameters**|Subnet-specific settings (tempo, max miners, min stake, etc.)|

Each subnet is essentially a **company with no employees** — it defines a job, posts a reward, and the market fills it.

---

## 2. Subnet Lifecycle

### Creation

- Subnet owner pays a **registration fee in TAO** to create a subnet
- The fee is burned (removed from supply) — this is deflationary
- Registration cost scales with network demand: more subnets = higher cost to create new ones
- Currently capped at **32 active subnets** on mainnet (SN0–SN32), though this is a governance parameter

### Operation

- Subnet owner deploys an **incentive mechanism** — the code that scores miner outputs
- Miners and validators register to the subnet (also pay a small TAO registration fee)
- Validators query miners, score outputs, submit weights `W`
- Yuma Consensus runs each **tempo** (a fixed number of blocks) to compute emissions

### Death

- If a subnet fails to attract miners/validators, it produces no useful intelligence
- Emissions continue to flow but rewards go unclaimed or concentrate in few participants
- Subnet owners have strong incentive to keep their subnet healthy or lose their 18% cut

---

## 3. Subnet Hyperparameters

Each subnet has configurable parameters set by the owner:

|Parameter|What it controls|
|---|---|
|`tempo`|How many blocks between each Yuma Consensus run (emission cycle)|
|`max_n`|Maximum number of registered neurons (miners + validators)|
|`min_allowed_weights`|Minimum number of miners a validator must score|
|`max_allowed_weights`|Maximum weights a validator can set|
|`immunity_period`|New miner grace period — protected from deregistration|
|`min_stake`|Minimum TAO stake required to register as validator|
|`adjustment_interval`|How often registration costs auto-adjust|

These parameters let subnet owners tune the competitive dynamics of their subnet — how many participants, how fast emissions cycle, how much stake validators need.

## 4. Emission Allocation Across Subnets

Not all subnets receive equal emissions. The global emission `τ` is split across subnets based on their **relative performance** as judged by the root network (Subnet 0).

### The Root Network (Subnet 0)

Subnet 0 is special — it is the **meta-subnet** that ranks all other subnets:

- Validators in Subnet 0 are the **top 64 validators by stake** across the entire network
- They assign weights to subnets (not miners) based on the subnet's value to the network
- Yuma Consensus runs on these weights to produce a **subnet ranking**
- Emissions are distributed proportionally to this ranking

$$\tau_{\text{subnet}_k} = \tau \cdot \frac{R_k}{\sum_j R_j}$$

- A subnet producing valuable, in-demand intelligence gets more emissions
- A stagnant or low-quality subnet gets less
- This creates **competition between subnets**, not just within them

### Emission Split Within a Subnet

Once a subnet receives its allocation `τₖ`:

|Recipient|Share|
|---|---|
|Subnet Owner|18%|
|Validators|41%|
|Miners|41%|

## 5. The Incentive Mechanism — The Subnet's Core

The **incentive mechanism** is the code the subnet owner writes and deploys. It defines:

1. **What task miners must perform** — what query they receive
2. **How outputs are evaluated** — the scoring function
3. **What score gets assigned** — the weight `wᵢⱼ` a validator should give miner `j`

### Example: A text generation subnet

- Validator sends a text prompt to all miners
- Miners return completions
- Incentive mechanism scores completions on coherence, relevance, fluency
- Validator submits weights based on these scores

### Example: A financial prediction subnet

- Validator sends a price prediction request
- Miners return predictions
- Incentive mechanism scores predictions against actual outcomes
- Best predictors earn highest weights

### The subnet owner's design challenge

The incentive mechanism must be:

- **Objective enough** that validators can't easily fake scores
- **Hard enough** that miners can't game it with cheap tricks
- **Clear enough** that good miners know what to optimize for
- **Verifiable enough** that Yuma Consensus can detect validator dishonesty

A poorly designed incentive mechanism will attract miners who game the metric rather than solve the actual problem — Goodhart's Law applied to AI.

## 6. Miner Registration and Deregistration

### Registering

Miners pay a TAO registration fee to join a subnet. This fee:

- Burns TAO (deflationary)
- Auto-adjusts based on subnet demand — popular subnets cost more to enter
- Creates a baseline commitment — you can't spam-register

### Immunity Period

New miners get an **immunity period** after registration during which they cannot be deregistered. This gives them time to spin up their model and start producing outputs.

### Deregistration — Pruning

When a subnet reaches `max_n` capacity and a new miner wants to register, the **lowest-ranked miner is pruned**:

$$\text{deregister: } \arg\min_i(R_i)$$

This is the Darwinian mechanism in action — underperformers are automatically ejected to make room for new entrants who might do better. No human decision required.

---

## 7. Validator Registration

- Validators must meet `min_stake` threshold to register in a subnet
- This is a **capital barrier** — it prevents sybil attacks (spinning up fake validators with zero stake)
- Validators also burn a registration fee
- A validator can operate across multiple subnets simultaneously — their stake is global, their weights are per-subnet

## 9. Subnet Competition Dynamics

### Miners choose subnets strategically

A miner with a strong LLM doesn't compete in an image generation subnet — they find the subnet where their model has an edge. This creates **specialization**:

- Miners concentrate where their skills are competitive
- Subnets attract the world's best models for their specific task
- Over time, each subnet develops a deep pool of specialized intelligence

### Validators follow stake opportunity

Validators go where emissions are highest relative to competition. If a subnet is undervalued (high emissions, few validators), sophisticated validators enter to capture that yield.

### Subnet owners compete for talent

Subnet owners must make their subnet attractive:

- High emission allocation (by producing valuable intelligence judged by SN0)
- Clear, fair incentive mechanisms
- Reasonable registration costs and hyperparameters

## 10. The Full Subnet Flow

```
Global emission τ
        ↓
Root Network (SN0) ranks subnets
        ↓
Each subnet receives τₖ proportional to rank
        ↓
Within subnet:
  Subnet Owner defines incentive mechanism
          ↓
  Validators query miners every tempo blocks
          ↓
  Miners return outputs
          ↓
  Incentive mechanism scores outputs
          ↓
  Validators submit weights W
          ↓
  Yuma Consensus → R, T, I
          ↓
  TAO distributed:
    18% → Subnet Owner
    41% → Validators (weighted by s · T)
    41% → Miners (weighted by R)
          ↓
  Lowest-ranked miners pruned if at capacity
```
